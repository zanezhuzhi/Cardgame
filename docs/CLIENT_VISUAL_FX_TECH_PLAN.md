# 客户端视觉与动效技术方案

> **定位**：核心玩法与同步逻辑稳定后，分阶段提升战斗与卡牌反馈的观感；**不更换**现有主技术栈（Vue 3 + Vite），通过**分层叠加**引入动效与粒子管线。  
> **参照风格**：2D 卡牌战斗反馈可参考《杀戮尖塔》层级——精灵/粒子/UI 动效，而非 3D 大世界。

---

## 一、现状与前提

| 项目 | 说明 |
|------|------|
| 主界面 | `client/src/App.vue` 为主，DOM + CSS，配合全局主题 `client/src/styles/theme-wafuu.css` |
| 构建 | Vite；可选 `build:electron` 发布桌面端 |
| 依赖 | `package.json` 已包含 `pixi.js`（^7.x），**源码中尚未作为主渲染循环使用**，可作为动效层基底 |

---

## 二、总体原则

1. **主栈保持 Vue + Vite**：规则状态、房间、Socket 仍以 Vue 组件与现有模块为准。  
2. **表现与逻辑解耦**：动效由「事件/状态快照」触发，不写死在业务判断里；便于单人/多人共用同一套视觉入口。  
3. **DOM 与 Canvas/WebGL 分层**：  
   - **DOM**：布局、表单、长文本、复杂列表、无障碍与输入法友好区域。  
   - **Canvas（Pixi）**：高密度绘制、粒子、短时全屏/局部特效、卡牌轨迹与打击感（可选逐步迁移「战场层」）。  
4. **默认不抢交互**：全屏或大面积特效层使用 `pointer-events: none`，仅在需要点选的局部开启点击。

## 三、动效实现分档（按需选用，可并存）

### 3.1 轻量（纯 CSS / SVG）

- **适用**：悬停高亮、禁用态、按钮反馈、简单光晕与过渡。  
- **成本**：无额外运行时；注意与全局主题的**特异性**（避免全局 `.hand-card.unplayable` 等覆盖 scoped 规则）。  
- **资源**：WebP/PNG/SVG，走 Vite 静态资源或 `public/`。

### 3.2 中量（Pixi 7 或 Canvas2D —— 推荐用于「尖塔感」）

- **适用**：卡牌移动/缩放轨迹、出牌/受击粒子、聚拢尘雾、Bitmap 飘字、简单滤镜（发光、闪白）。  
- **与尖塔类比**：尖塔为 libGDX 2D；网页侧 **Pixi 足以覆盖同类 2D 反馈**，无需为达此目标引入 Three.js。  
- **可选增强**：`@pixi/particle-emitter` 等（按包体与许可评估后引入）。  

### 3.3 重量序列资源（与引擎独立）

- **Lottie**：适合 UI 条、装饰性过场（JSON + 设计师导出）。  
- **Spine / DragonBones 等 2D 骨骼**：适合固定演出；需额外运行时与导出工作流。  
- **选用建议**：强依赖「一条成片动画」时用序列工具；零碎战斗反馈仍宜用粒子 + 补间。

## 四、推荐架构（Vue + Pixi 叠加）

```
┌─────────────────────────────────────────┐
│  Vue 根布局（现有 App / Router）          │
│  ┌─────────────────────────────────────┐ │
│  │ DOM：牌桌 UI、手牌区、日志、弹窗     │ │
│  └─────────────────────────────────────┘ │
│  ┌─────────────────────────────────────┐ │
│  │ 特效层（建议 Teleport → body）        │ │
│  │  固定定位 + z-index 高于战场背景      │ │
│  │  单例 Pixi Application + 容器 div     │ │
│  │  pointer-events: none（默认）         │ │
│  └─────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

- **状态同步**：Vue 侧在「打出卡牌」「造成伤害」「式神技能」等节点 `emit` 或由 Store 提交**视觉事件**（含屏幕坐标或槽位 id）；Pixi 层仅订阅并播放，播完自销毁粒子/显示对象。  
- **坐标映射**：若卡牌仍在 DOM 上，可用 `getBoundingClientRect()` 把 DOM 矩形转为画布内坐标，再播放飞行轨迹。

## 五、资源与工程

| 事项 | 建议 |
|------|------|
| 体积 | 纹理打图集（Sprite Sheet）、按需 `import()` 关卡/主题资源 |
| 路径 | Web 与 Electron 统一资源基路径；避免硬编码仅适用于 dev 的 URL |
| 预加载 | 进战斗前预加载本局用到的纹理与配置，避免首帧卡顿 |
| 降级 | 设置「低画质」：减少粒子数、降低滤镜、关闭全屏后处理 |

## 六、性能与体验

- **dpr**：`devicePixelRatio` 设上限（如 2），避免 4K 下分辨率爆炸。  
- **Ticker**：与浏览器帧率对齐；后台标签页可暂停 Pixi ticker。  
- **热更开发**：动效层故障应可整体关闭（开关或环境变量），保证策划仍能用「丑但可用」版测逻辑。

---

## 七、落地阶段建议（逻辑完成后再做）

1. **P0**：统一视觉事件命名与 payload（与 `EffectEngine`/服务端日志可对齐，便于调试）。  
2. **P1**：接入 Pixi 全屏（或手牌区上方）透明层 + 1～2 个标准特效（出牌、受击）。  
3. **P2**：卡牌轨迹与槽位对齐；粒子库与调色板与 `策划文档/美术规范.md` 一致。  
4. **P3**：低画质模式、资源分包与预加载策略。
