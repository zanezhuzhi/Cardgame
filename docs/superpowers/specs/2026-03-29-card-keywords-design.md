# 设计说明：卡牌关键词规范、数据贯通与三味结算

> **状态**：已定稿（策划确认：鬼火牌 = 关键词含「鬼火」；实现方案 A）  
> **日期**：2026-03-29  
> **关联**：`策划文档/游戏规则说明书.md`、`策划文档/卡牌数据/妖怪卡.md`、`shared/data/cards.json`、`shared/types/cards.ts`、`shared/data/loader.ts`、`server/src/game/MultiplayerGame.ts`（三味）

---

## 1. 背景与问题

- 妖怪卡表「关键词」列存在 **御魂 / 鬼火 / 妨害** 等多标签（`/` 分隔），与效果中「『鬼火』牌」「『御魂』」等表述绑定。
- **运行时** `CardInstance` 若未携带关键词，则 **三味**（「每使用了 1 张『鬼火』牌或阴阳术，则伤害 +2」）无法识别 **薙魂** 等已在 `cards.json` 中标记为 `御魂/鬼火` 的牌。
- 根因：**`createYokaiInstance` 等路径未把 `subtype` 解析并写入 `CardInstance.keywords`**；部分逻辑误依赖未统一的 `tags` / `subtype` 字段。

---

## 2. 规则定义（策划向）

### 2.1 关键词是什么

- **关键词**是卡牌上用于 **分类与效果引用** 的标签集合。
- **来源**：以 **卡牌数据**（`cards.json` 中游荡妖怪的 `subtype` 字符串，形如 `御魂/鬼火`）为 **实现单一事实来源**；`妖怪卡.md` 表格「关键词」列与之 **对齐维护**。
- **解析**：将 `subtype` 按 **`/`** 拆成若干非空片段，去掉首尾空格，得到 **`keywords: string[]`**（顺序可按表内书写顺序，不要求排序）。

### 2.2 「鬼火」牌（封闭式定义）

- **『鬼火』牌** = **`cardType === 'yokai'`** 且该牌实例的 **`keywords` 中含字符串 `'鬼火'`**。
- **不** 以「效果是否产生鬼火资源」为准；无「鬼火」关键词则 **不是** 三味规则意义上的鬼火牌。

### 2.3 其它引用

- 效果文本中的 **「御魂」「妨害」** 等，若无单独说明，**以对应关键词是否包含该标签为准**（与现有策划表一致）。
- **阴阳术**：`cardType === 'spell'`，**不** 使用妖怪关键词；三味中与鬼火牌 **并列** 统计。

---

## 3. 技术方案（方案 A）

### 3.1 规范字段

- **权威结构**：`CardInstance.keywords?: string[]`（已存在于 `shared/types/cards.ts`）。
- **单一解析函数**（建议放在 `shared/data/loader.ts` 或紧邻的 `shared/game/cardKeywords.ts`）：
  - 输入：`YokaiCard` 或 `subtype` 字符串 + 可选显式 `keywords`（若未来 JSON 增加顶层 `keywords` 数组，则 **优先生效** 或与 `subtype` 合并策略在实现时二选一并写入 spec 子条款）。
  - 输出：规范化 `string[]`。

### 3.2 实例化

- **`createYokaiInstance`**：**必须** 填充 **`keywords`**（从 `subtype` 解析）。
- **所有** 构造 `CardInstance` 的路径（**单机客户端** `SinglePlayerGame`、测试 harness、极少数手搓对象）应 **复用同一逻辑** 或从完整 `YokaiCard` 生成，避免双轨。

### 3.3 三味（服务端）

- 统计本回合 **已使用** 的：
  - 每张 **阴阳术**（`spell`），与
  - 每张 **`keywords` 含 `'鬼火'`** 的 **御魂**（`yokai`）。
- **计数口径**：与当前「本回合打出 / `played`」的语义一致，**不重复计数**；**移除或改写** 对未定义字段 **`tags`** 的依赖。
- 建议抽取 **`cardHasKeyword(card, '鬼火')`** 供三味、幽谷响 AI 评分、及未来效果复用。

### 3.4 客户端展示（建议第二阶段）

- 卡牌详情 / 标签栏：展示 **完整 `keywords`**（多标签），避免仅显示「御魂」丢失「鬼火」。可与服务端修复 **分 PR** 交付。

## 5. 测试与验收

- **集成 / 单元**：本回合打出 **薙魂**（或另一张确认含鬼火关键词的御魂）后，**三味** 结算伤害增量与「每使用一张鬼火牌 +2」一致；**无鬼火关键词**的御魂不计入。
- **回归**：依赖 `keywords` 的其它逻辑（如幽谷响 AI 使用 `keywords`）在实例补全后行为 **不变或更符合设计**。
