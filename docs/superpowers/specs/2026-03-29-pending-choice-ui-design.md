# 设计说明：统一 Pending Choice 外壳与结算中部提示

> **状态**：已定稿（产品与策划确认，2026-03-29）  
> **实现策略**：混合方案（服务端给出「谁看提示、用哪种计时」；客户端负责外壳排版与展开牌面全文）  
> **关联规则**：《游戏规则说明书》§5.5；§5.4（回合外 5 秒、行动限时暂停）

---

## 1. 背景与目标

- 各类 `pendingChoice`（不论本回合玩家还是回合外被点名）需要**统一认知锚点**：来源牌、当前在做什么、还剩多少时间。
- 多效果卡连续结算时，若直接进入第二段选择，玩家易困惑；需要与**信息栏同源**的**中部淡化提示**，在关键步骤切换时播放。

---

## 2. 已定决策摘要

| 项 | 结论 |
|----|------|
| 外壳布局 | 左：卡牌来源；右上：默认「当前步骤」短说明 + **可展开**该牌**完整规则原文**；右下：倒计时条；下方：各 type 既有交互主体 |
| 右下计时 | **当前行动玩家**且交互由本人完成时：与§5.2 行动阶段**全局回合计时**同一数据源（含仅 1 人时「无限制」的展示约定）。**回合外被点名**（§5.4）：**独立 5 秒**响应倒计时，仅表示本段反馈，不与全局条混用 |
| 中部提示受众 | **自己打牌结算**：仅**本人**中部提示。**妨害等强制回合外反馈**：**被点名玩家**必须中部提示；**其余玩家**默认仅信息栏，不弹中部 |
| 技术路线 | **混合**：计时模式与 toast 受众以服务端状态/事件为权威；展开全文由客户端查卡牌数据 |

---

## 3. 服务端与协议（建议）

在现有 `pendingChoice`（或并行事件流）上扩展**展示元数据**（具体字段名实现阶段敲定），至少区分：

- `timerMode`: `turnTotal` | `offTurnResponse`（后者对应 5 秒窗口，与 §5.4 一致）
- `toast`（可选）：`messageId` 或与信息栏**完全相同的文案**、以及 `recipientPlayerIds: string[]`（仅列名需要中部提示的座位）

原则：**信息栏与中部提示共用同一文案来源**（同一次事件或同一 `messageId`），避免两处不一致。

---

## 4. 客户端

- **统一外壳组件**包裹现有各 pending 弹层：只增加布局与计时绑定，不拆分各业务 type 的选项逻辑。
- **右上**：默认短文案来自服务端 `stepSummary`（或等价字段）；展开区读本地卡牌定义中的完整规则文本。
- **左**：来源卡缩略 + 名称；无单卡来源时约定占位文案（如系统/妨害类标题）。
- **中部提示**：订阅与信息栏相同事件；按 `recipientPlayerIds` 过滤，仅命中本地玩家时播放；动画为短显+淡出，不阻塞操作（具体时长实现可调，建议 ~2–3s）。

---

## 5. 边界

- **超时默认**：回合外 5 秒与§5.4 默认策略一致；外壳须显示可读倒数。
- **多卡触发**：协议若仅携带「主来源」则 UI 只展示主来源；若后续需多卡再扩展字段。

---

## 6. 验收测试（建议）

1. 单卡两效果：第一段结算后出现中部提示，再进入第二段 pending；外壳左上/右上信息正确。
2. 本回合 pending：右下与游荡妖怪上方全局计时一致。
3. 妨害点名：被点名玩家见 5 秒条 + 中部提示；其他玩家仅信息栏。
4. 等待他人时：§5.4 行动阶段暂停；与被点名玩家 5 秒条语义不冲突（全局暂停 vs 个人反馈窗）。
