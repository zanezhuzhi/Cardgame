## 一、核心规则（术语）

撰写《卡牌具体文档》与实现妨害、互动类效果时，若无另行定义：

- **【对手】** 意为 **【除自己以外的其他玩家】**，**包括 AI 对手**（由 AI 代打的玩家座位与真人玩家同等视为「对手」）。
- 文案中「每名对手」「所有对手」等均以上述定义为准，不因操控者为真人或 AI 而缩小范围。
- **【弃置】**：将卡牌从牌库/手牌**直接**移入弃牌堆（**不经过已打出区**）的操作。规则文本中出现的【弃置】均按 **【主动弃置】** 判定，**不包括**回合结束清理阶段的规则弃置。完整释义见 [游戏规则说明书 §2.3](../游戏规则说明书.md#23-效果用语批注)。

### 妖怪关键词与 `subtype`（实现约定）

- 游荡妖怪在 `shared/data/cards.json` 中使用 **`subtype`** 字符串记载类型标签（与《妖怪卡》表「关键词」列一致，多标签用 `/` 分隔）。
- 生成 **`CardInstance`** 时必须写入 **`keywords: string[]`**（与共享逻辑 `resolveYokaiKeywords` / `parseSubtypeToKeywords` 一致），**禁止**在三味、幽谷响 AI 等逻辑中依赖未赋值的 `tags` 或实例上缺失的 `subtype` 做关键词判定。
- 「『鬼火』牌」等引用以 **`cardType === 'yokai'` 且 `keywords` 含对应串** 为准；阴阳术单独按 `spell` 统计。策划说明见 [《游戏规则说明书》2.4](../游戏规则说明书.md#24-关键词与类型标签)；技术说明见 `docs/superpowers/specs/2026-03-29-card-keywords-design.md`。

### 妨害与反妨害（铮、青女房）

撰写文档与实现「是否算受到【妨害】/ 是否可走妨害抵抗（铮、青女房等）」时：

- **妨害来源不限于游荡御魂**：**式神技能**文案中若定义为 **[妨害]**（或与技能引擎 `isHarassment` 等等价标记一致），即属于妨害类效果；实现时不能只依赖妖怪 `CardInstance` 的 `keywords`，须与式神技能定义一并考虑。
- **铮、青女房是反妨害牌**：用于抵消或豁免**他人发起的妨害**，自身**不是**妨害发起方；数据中 **铮** 为 **御魂/反制**、**青女房** 为 **御魂/鬼火**，**不应**带有发起妨害用的关键字「妨害」。
- **判定「是否含关键字妨害」（发起侧）**：
  - **游荡御魂**：以实例的 `keywords` / `subtype` 解析结果为准（与上节「妖怪关键词」约定一致）；当前数据中带 **御魂/妨害** 的妖怪见《妖怪卡》表。
  - **飞缘魔**：为**复制**当前鬼王御魂效果；是否视为关键字妨害，以**当次被执行的鬼王御魂**（鬼王魂对应的卡牌定义）是否含「妨害」为准，**不以飞缘魔本体的 subtype 为准**。
  - **镇墓兽**（左手边指定「本回合不可退治某目标」）：属**规则性限制**（禁止退治），**不是【妨害】**；不与「受到妨害 → 铮/青女房」的触发口径混用；实现上亦不应与妨害管线混为一谈。

### 抓牌与获得的区分

| 术语 | 定义 | 来源 | 目标位置 |
|------|------|------|----------|
| **【抓牌】** | 从**自己的牌库**抽取卡牌 | 己方牌库 | **手牌** |
| **【获得】** | 从**公共牌堆**转化为"我的牌"（牌库数量增加） | 公共牌堆（如恶评牌库） | **弃牌堆**（默认） |

**示例**：
- 「抓牌+1」：从自己牌库顶抽 1 张入**手牌**
- 「获得 1 张恶评」：从恶评牌库抽 1 张加入自己的**弃牌堆**

**注意**：
- 恶评卡、令牌等可以存在于手牌中（如通过其他效果），但**无法被使用**
- 若效果明确指定目标位置（如"加入手牌"），则按效果描述执行

### 单目标效果与可用性

- **适用范围**：卡牌或技能的效果**唯一**依赖「选择 1 个合法目标」才能完整结算，且**没有**无目标分支时，视为**纯单目标**效果。
- **规则**：当前局面下若**不存在任何合法目标**，则该卡**不可打出**，该技能**不可发动**。
- **合法目标**：以牌面/文档约定的条件为准（如生命值区间、区域等）。
- **示例（天邪鬼绿）**：场上**没有**满足条件的可退治妖怪时，**不可使用该牌**。

### 手牌上限

- **上限值**：**10 张**
- **检查时机**：每次**抓牌前**
- **规则**：若抓牌前手牌数量 **≥ 10**，则**跳过本次抓牌**

**示例（树妖效果：抓牌+2，弃置1张）**：
1. 当前 10 张手牌 → 使用树妖（剩 9 张）
2. 抓第 1 张 → 10 张
3. 抓第 2 张 → 检查 = 10，**跳过** → 仍是 10 张
4. 弃置 1 张 → 9 张

**实现说明**：
- 服务端 `drawCards` 函数在每次抽牌循环中检查 `player.hand.length >= 10`
- 若满则打印日志并跳出循环
- 后续抓牌操作不会补回被跳过的牌

## 二、公共效果规则

### 牌库展示

> 定义「查看牌库」效果的统一规则与UI表现

**基础规则**：
1. 所有玩家默认**不能查看**自己或他人的牌库内容
2. 当效果使牌库中某张牌被「展示」后，该信息将被记录
3. 展示信息为**私密信息**，仅**效果触发者**与**牌库拥有者**可见
4. 展示信息持续到该牌**离开原位置**（被抽走、弃置、超度等）

**UI表现**：

| 场景 | 牌库区域显示 |
|------|-------------|
| 无已展示牌 | 牌堆图标 + 剩余数量（不可点击） |
| 有已展示牌 | 牌堆图标 + 剩余数量（**可点击**，高亮提示） |
| 点击后弹窗 | 已展示的牌（正面） + 未展示的牌（背面 + 数量） |

**涉及卡牌**：唐纸伞妖、天邪鬼黄、赤舌、魅妖、木魅、幽谷响

## 五、强制中止与AI接管

### 5.1 强制中止行为

当效果执行过程中被外部因素强制中断时的默认行为：

| 选择类型 | 默认行为 |
|----------|----------|
| 二选一 | 选择第一个选项 |
| 选择手牌 | 不选择任何牌，若必须选则选第一张 |
| 选择目标 | 选择第一个合法目标 |

**事务性原则**：
- 已生效的数值变更不回滚
- 未完成的选择步骤执行默认策略

### 5.2 AI策略等级

| 等级 | 名称 | 说明 |
|:----:|------|------|
| L0 | 随机 | 随机选择（仅用于测试） |
| L1 | 规则 | 按固定规则选择（**当前使用**） |
| L2 | 启发 | 考虑当前局面综合判断（未来扩展） |

### 7.1 问题分类速查

| 类型 | 描述 | 自查点 |
|------|------|--------|
| **逻辑不一致** | 共享层与服务端逻辑不一致 | 修改效果时**必须同步检查**两处 |
| **交互未触发** | 效果需要玩家选择，但直接执行了默认行为 | 涉及选择的效果**必须等待玩家响应** |

| **响应链断裂** | 触发了交互但缺少响应处理 | 新增交互类型时**前后端必须同步** |

### 7.3 轮入道兼容检查

> **轮入道** 是游戏中最复杂的御魂，可弃置 1 张 HP≤6 的御魂并**执行其效果 2 次**。  
> 所有新开发的御魂都必须考虑被轮入道触发的场景。

#### 轮入道机制概述

| 概念 | 说明 |
|------|------|
| **队列执行** | 轮入道通过 `wheelMonkQueue` 队列管理效果执行，每次执行完整效果（包含交互选择） |
| **执行次数** | `remainingExecutions: 2`，每完成一次完整效果递减 |
| **交互等待** | 若效果触发 `pendingChoice`，队列等待响应后再继续下一次执行 |

#### 开发检查清单

**【步骤1】效果分类判断**

| 效果类型 | 示例 | 轮入道处理方式 |
|----------|------|----------------|
| **纯数值效果** | 抓牌+1、伤害+2、鬼火+1 | 在 `executeYokaiEffect` 的对应 `case` 中处理；轮入道通过 `multiplier` 翻倍 |
| **自己交互效果** | 树妖（抓2选弃1）、日女巳时（三选一） | 在 `executeYokaiEffect` 中添加完整 `case`，触发 `pendingChoice` |
| **对手交互效果** | 赤舌（对手选择）、魅妖（选对手牌） | 同上，完整执行交互流程，超时时自动AI策略 |
| **妨害无交互** | 雪幽魂（对手获得恶评） | 在 `executeYokaiEffect` 中处理，效果自动对所有对手执行 |

**【步骤2】代码实现要点**

1. **有交互的御魂**：
   ```typescript
   // 在 executeYokaiEffect 中添加 case
   case '新御魂名': {
     // 执行效果逻辑
     // 设置 pendingChoice 等待玩家响应
     this.state.pendingChoice = {
       type: 'newYokaiChoice',  // 新增类型需同步到 server/src/types/index.ts
       playerId: player.id,
       prompt: '选择...'
     };
     break;
   }
   ```

2. **响应处理函数**：
   ```typescript
   // 在响应处理函数末尾添加队列继续调用
   public handleNewYokaiChoiceResponse(...): { success: boolean; error?: string } {
     // ... 处理响应逻辑 ...
     
     // 清除等待状态
     this.state.pendingChoice = undefined;
     
     // ⚠️ 关键：检查是否需要继续轮入道队列
     this.checkAndContinueWheelMonkQueue();
     
     this.notifyStateChange({ type: 'STATE_UPDATE', state: this.state });
     return { success: true };
   }
   ```

**【步骤3】测试验证**

- [ ] 单独使用该御魂，效果正常
- [ ] 轮入道+该御魂，效果执行2次
- [ ] 轮入道+该御魂，每次交互都正确弹出选择框
- [ ] 轮入道+该御魂，超时时自动按AI策略处理

## 八、Bug 修复规范 ⚠️

> **核心原则**：每个发现的 bug 都必须有对应的自动化测试用例，防止回归。

### 8.1 Bug 修复流程

| 步骤 | 操作 | 必须 |
|:----:|------|:----:|
| **1** | 复现 Bug（使用测试 GUI 或命令行） | ✅ |
| **2** | 定位问题代码 | ✅ |
| **3** | **先写/更新测试用例**（TDD） | ⚠️ **必须** |
| **4** | 修复代码 | ✅ |
| **5** | 运行测试确认通过 | ✅ |
| **6** | 提交并验证其他测试不受影响 | ✅ |

### 8.2 测试用例编写规范

**位置**：`shared/game/effects/*.test.ts`

**命名格式**：
```
describe('卡牌名/功能名', () => {
  it('🟢 正常流程描述', ...);  // 正常情况
  it('🔴 边界条件描述', ...);  // 异常/边界
});
```

**必须覆盖**：
- ✅ 正常执行路径
- ✅ 边界条件（空牌库、鬼火不足、手牌为空等）
- ✅ 交互选择的各种分支（选择A、选择B、超时默认）
- ✅ 组合场景（如轮入道触发其他御魂）

### 8.3 Bug 示例记录

> 便于追溯和避免类似问题

| 日期 | Bug 描述 | 根因分析 | 修复方案 | 测试用例 |
|------|----------|----------|----------|----------|

### 8.5 测试未通过时的处理

1. **不要忽略**：每个失败的测试都表示潜在问题
2. **不要 `skip`**：除非明确标记为"待实现"功能
3. **不要手动验证代替**：自动化测试是持续集成的基础
4. **求助渠道**：将测试输出复制给 AI（使用"复制失败提示"按钮）
