# 卡牌开发流程规范

> 所有卡牌效果开发必须遵循此流程

---

## 核心规则（术语）

撰写《卡牌具体文档》与实现妨害、互动类效果时，若无另行定义：

- **【对手】** 意为 **【除自己以外的其他玩家】**，**包括 AI 对手**（由 AI 代打的玩家座位与真人玩家同等视为「对手」）。
- 文案中「每名对手」「所有对手」等均以上述定义为准，不因操控者为真人或 AI 而缩小范围。
- **【弃置】**：将卡牌从牌库/手牌**直接**移入弃牌堆（**不经过已打出区**）的操作。规则文本中出现的【弃置】均按 **【主动弃置】** 判定，**不包括**回合结束清理阶段的规则弃置。完整释义见 [游戏规则说明书 §2.3](../游戏规则说明书.md#23-效果用语批注)。

| 类型 | 对应文档 |
|------|----------|
| 妖怪/御魂 | [妖怪卡.md](./妖怪卡.md) |
| 鬼王 | [鬼王卡.md](./鬼王卡.md) |
| 式神 | [式神卡.md](./式神卡.md) |
| 阴阳术 | [阴阳术卡.md](./阴阳术卡.md) |
| 恶评 | [恶评卡.md](./恶评卡.md) |

### 1.2 查找或创建卡牌文档

1. **查找现有文档**：检查 `策划文档/卡牌数据/卡牌具体文档/` 下是否已有对应文件  
2. **创建新文档**：如不存在，在该目录下创建；文件名与 `shared/data/cards.json` 中 `id` 一致：`yokai_XXX_名称.md`、`token_XXX_名称.md`
3. **关联到牌表**：将新文档路径添加到对应类型的汇总文档末尾
4. **⚠️ 更新索引**：创建/修改文档后，必须更新本文档 [§五、已开发卡牌索引](#五已开发卡牌索引) 中的对应条目

## 二、文档内容规范

每张卡牌的 `.md` 文档应包含以下章节：

### 📋 基础信息

```markdown
## 📋 基础信息

| 属性 | 值 |
|------|---|
| 名称 | xxx |
| HP | x |
| 声誉 | x |
| 类型 | 御魂/伤害/抓牌/鬼火/声誉/妨害 |
| 总数量 | N（cards.json中的count字段） |
| 多人标记 | M（cards.json中的multiPlayer字段） |
| 初始张数 | N-M（标准模式3-4人使用的数量） |
```

#### 多人标记规则

> 参考《游戏规则说明书 §4.2》：🔷纸人标记表示该卡种中有几张是5-6人多人专属

| 计算公式 | 说明 |
|----------|------|
| **标准模式（3-4人）** | 使用 `count - multiPlayer` 张 |
| **多人模式（5-6人）** | 使用全部 `count` 张 |

**示例**：

| 卡牌 | count | multiPlayer | 标准模式 | 多人模式 |
|------|:-----:|:-----------:|:--------:|:--------:|
| 赤舌 | 2 | 0 | 2张 | 2张 |
| 唐纸伞妖 | 2 | 1 | 1张 | 2张 |
| 镜姬 | 2 | 2 | 0张（不使用） | 2张 |

> ⚠️ `multiPlayer = count` 时，该卡牌**仅在多人模式使用**

### 📜 效果原文

```markdown
## 📜 效果原文

> 摘自牌面/说明书的原始效果描述
```

### 🔍 效果分解

```markdown
## 🔍 效果分解

### 步骤拆解
1. 步骤1描述
2. 步骤2描述

### 关键规则
- 规则1
- 规则2
```

### 🎲 AI 行为决策

```markdown
## 🎲 AI 行为决策

| 决策点 | AI策略 | 优先级说明 |
|--------|--------|-----------|
| `onChoice` | 选择收益最大的选项 | 伤害 > 抓牌 > 鬼火 |
| `onSelectCards` | 选择价值最低的牌 | 按HP升序 |
| `onSelectTarget` | 选择HP最低的目标 | 同HP时随机 |
```

### ⚔️ 交互场景

```markdown
## ⚔️ 交互场景

### 场景1：正常使用
游荡区: [心眼(HP5), 天邪鬼青(HP3)]
...
```

### ✅ 验收标准

```markdown
## ✅ 验收标准

- [ ] 基础测试条件
- [ ] 边界测试条件
- [ ] 强制中止：超时时效果按默认策略正确结算
- [ ] AI接管：AI玩家能正确执行此效果
```

### 🛠️ 实现备注（可选）

```markdown
## 🛠️ 实现备注

### 涉及文件
- `xxx.ts` - 描述

### 注意事项
- 注意点1
```

## 三、强制中止与AI接管

### 3.1 强制中止行为

> 定义效果执行过程中被外部因素强制中断时的默认行为

| 选择类型 | 默认行为 |
|----------|----------|
| `onChoice`（二选一） | 选择第一个选项（index=0） |
| `onSelectCards`（选择手牌） | 不选择任何牌，若必须选则选第一张 |
| `onSelectTarget`（选择目标） | 选择第一个合法目标 |

**事务性原则**：
- 已生效的数值变更不回滚
- 未完成的选择步骤执行默认策略
- 日志标注 `[超时]` 或 `[中止]`

### 3.2 AI策略等级

| 等级 | 名称 | 说明 |
|:----:|------|------|
| L0 | 随机 | 随机选择（仅用于测试） |
| L1 | 规则 | 按固定规则选择（**当前使用**） |
| L2 | 启发 | 考虑当前局面综合判断（未来扩展） |

## 四、开发执行（6步TDD流程）

| 步骤 | 内容 |
|:----:|------|
| **1** | 实现功能逻辑 |
| **2** | 定义验证点 |
| **3** | 编写TDD用例 |
| **4** | 运行 `npm test` |
| **5** | 创建GM指令 |
| **6** | 提交并重启 |

### GM 指令速查

| 指令 | 说明 | 示例 |
|------|------|------|
| `/addcard <卡牌名> [数量]` | 添加卡牌到手牌 | `/addcard 赤舌` |
| `/status` | 查看房间状态 | `/status` |
| `GET /api/gm/addcard/:roomId/:playerId/:cardName/:count` | HTTP添加卡牌 | - |
