# 卡牌开发流程规范

> 所有卡牌效果开发必须遵循此流程

---

## 一、开发前准备

### 1.1 确定卡牌类型

根据要开发的卡牌，确定其所属类型：

| 类型 | 对应文档 |
|------|----------|
| 妖怪/御魂 | [妖怪卡.md](./妖怪卡.md) |
| 鬼王 | [鬼王卡.md](./鬼王卡.md) |
| 式神 | [式神卡.md](./式神卡.md) |
| 阴阳术 | [阴阳术卡.md](./阴阳术卡.md) |
| 恶评 | [恶评卡.md](./恶评卡.md) |

### 1.2 查找或创建卡牌文档

1. **查找现有文档**：检查 `D:\Cardgame\策划文档\卡牌数据\` 目录下是否存在该卡牌的 `.md` 文件
2. **创建新文档**：如不存在，在该目录下创建 `【卡牌名称】.md`
3. **关联到牌表**：将新文档路径添加到对应类型的汇总文档（妖怪卡.md / 鬼王卡.md 等）末尾

---

## 二、需求分析（文档内容）

每张卡牌的 `.md` 文档应包含以下内容：

### 2.1 基础信息

```markdown
## 基础信息

| 属性 | 值 |
|------|---|
| 名称 | xxx |
| HP | x |
| 声誉 | x |
| 类型 | xxx |
| 效果 | xxx |
| 多人标记 | 0/1 |
```

### 2.2 效果分解

将卡牌效果拆解为独立的逻辑步骤：

```markdown
## 效果分解

1. 步骤1描述
2. 步骤2描述
3. ...
```

### 2.3 模式实现分析

```markdown
## 单人模式实现

- 逻辑描述
- 涉及模块
- 风险点

## 多人模式实现

- 逻辑描述
- 涉及模块
- 交互机制
- 风险点
```

### 2.4 代码设计

```markdown
## 代码设计

### 涉及文件
- `xxx.ts` - 描述

### 关键函数
- `functionName()` - 描述

### 参数说明
- `param1` - 描述

### 注意事项
- 注意点1
- 注意点2
```

### 2.5 验收标准

```markdown
## 验收标准

- [ ] 条件1
- [ ] 条件2
- [ ] 边界情况
```

## 三、开发执行（6步TDD流程）

| 步骤 | 名称 | 内容 |
|:----:|------|------|
| **1** | **编写代码** | 实现功能逻辑（类型定义、核心方法、UI渲染） |
| **2** | **创建核心验证内容** | 定义验证点（边界条件、预期行为、错误处理） |
| **3** | **建立 Red/Green TDD 用例** | 编写测试用例，先失败(Red)后通过(Green) |
| **4** | **自动化测试** | 运行 `npm test` 直至全部通过 |
| **5** | **创建 GM 指令** | 添加调试指令协助手工测试 |
| **6** | **提交并重启** | Git提交 + 重启服务端/客户端，人工跟进测试 |

---

## 四、文档首次创建确认

若卡牌文档为**首次创建**，需经过人工确认后再进入开发阶段：

1. 展示完整的需求分析文档
2. 确认逻辑无误
3. 确认风险已评估
4. 获得确认后开始编写代码

若文档**已存在**，直接读取并进入开发执行阶段。
