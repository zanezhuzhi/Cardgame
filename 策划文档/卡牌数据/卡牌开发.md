# 卡牌开发流程规范

> 所有卡牌效果开发必须遵循此流程

---

## 一、开发前准备

### 1.1 确定卡牌类型

根据要开发的卡牌，确定其所属类型：

| 类型 | 对应文档 |
|------|----------|
| 妖怪/御魂 | [妖怪卡.md](./妖怪卡.md) |
| 鬼王 | [鬼王卡.md](./鬼王卡.md) |
| 式神 | [式神卡.md](./式神卡.md) |
| 阴阳术 | [阴阳术卡.md](./阴阳术卡.md) |
| 恶评 | [恶评卡.md](./恶评卡.md) |

### 1.2 查找或创建卡牌文档

1. **查找现有文档**：检查 `D:\Cardgame\策划文档\卡牌数据\` 目录下是否存在该卡牌的 `.md` 文件
2. **创建新文档**：如不存在，在该目录下创建 `【卡牌名称】.md`
3. **关联到牌表**：将新文档路径添加到对应类型的汇总文档（妖怪卡.md / 鬼王卡.md 等）末尾

## 二、需求分析（文档内容）

每张卡牌的 `.md` 文档应包含以下内容：

### 2.1 基础信息

```markdown
## 基础信息

| 属性 | 值 |
|------|---|
| 名称 | xxx |
| HP | x |
| 声誉 | x |
| 类型 | xxx |
| 效果 | xxx |
| 多人标记 | 0/1 |
```

### 2.2 效果分解

将卡牌效果拆解为独立的逻辑步骤：

```markdown
## 效果分解

1. 步骤1描述
2. 步骤2描述
3. ...
```

### 2.3 模式实现分析

```markdown
## 单人模式实现

- 逻辑描述
- 涉及模块
- 风险点

## 多人模式实现

- 逻辑描述
- 涉及模块
- 交互机制
- 风险点
```

### 2.4 代码设计

```markdown
## 代码设计

### 涉及文件
- `xxx.ts` - 描述

### 关键函数
- `functionName()` - 描述

### 参数说明
- `param1` - 描述

### 注意事项
- 注意点1
- 注意点2
```

### 2.5 强制中止行为

> 定义效果执行过程中被**外部因素强制中断**时的默认行为（如回合限时到期、玩家断线、服务器重启等）。

```markdown
## 强制中止行为

| 中止时机 | 已执行的部分 | 默认处理 |
|----------|------------|----------|
| 步骤N之前 | 无 | 效果整体跳过，卡牌正常进入弃牌堆 |
| 步骤N等待选择时 | 步骤1~N-1已生效 | 已生效部分保留，当前选择执行默认策略，后续步骤跳过 |
| ... | ... | ... |

### 默认选择策略

对于需要玩家选择的步骤，强制中止时按以下优先级执行：

| 选择类型 | 默认行为 |
|----------|----------|
| `onChoice`（二选一/多选一） | 选择第一个选项（index=0） |
| `onSelectCards`（选择手牌） | 不选择任何牌（空数组），若必须选则选第一张 |
| `onSelectTarget`（选择目标） | 选择第一个合法目标 |
| 对手的选择 | 自动选择第一个合法选项 |

### 事务性原则

- **已生效的数值变更不回滚**（伤害+N、鬼火+N、抓牌+N 等立即生效的操作）
- **未完成的选择步骤执行默认策略**，不跳过
- **日志记录**：标注 `[超时]` 或 `[中止]`，便于复盘
```

### 2.6 AI接管行为

> 定义 AI 玩家执行此效果、或人类玩家挂机由 AI 代打时的**决策逻辑**。

```markdown
## AI接管行为

### 触发条件

| 场景 | 说明 |
|------|------|
| AI玩家 | 游戏创建时指定为AI的玩家，始终由AI决策 |
| 挂机接管 | 人类玩家超过N秒未操作，临时切换为AI代打 |
| 断线接管 | 玩家断开连接后，由AI代替其完成当前回合 |

### AI决策表

对于此卡牌效果中的每个决策点，定义AI的选择策略：

| 决策点 | AI策略 | 优先级说明 |
|--------|--------|-----------|
| `onChoice` | 选择收益最大的选项 | 伤害 > 抓牌 > 鬼火（默认优先级） |
| `onSelectCards` | 选择价值最低的牌 | 按HP升序，同HP按声誉升序 |
| `onSelectTarget` | 选择HP最低的目标 | 同HP时随机 |
| 被妨害时的选择 | 选择损失最小的选项 | 弃置价值最低的牌 |

### AI策略等级

| 等级 | 名称 | 说明 |
|:----:|------|------|
| L0 | 随机 | 随机选择合法选项（仅用于测试） |
| L1 | 规则 | 按固定规则选择（默认，上方决策表） |
| L2 | 启发 | 考虑当前局面综合判断（未来扩展） |

> ⚠️ 当前阶段统一使用 **L1 规则策略**，每张卡牌文档中只需填写 L1 对应的决策表。
```

### 2.7 验收标准

```markdown
## 验收标准

- [ ] 条件1
- [ ] 条件2
- [ ] 边界情况
- [ ] 强制中止：超时时效果按默认策略正确结算
- [ ] AI接管：AI玩家能正确执行此效果
```

## 三、开发执行（6步TDD流程）

| 步骤 | 名称 | 内容 |
|:----:|------|------|
| **1** | **编写代码** | 实现功能逻辑（类型定义、核心方法、UI渲染） |
| **2** | **创建核心验证内容** | 定义验证点（边界条件、预期行为、错误处理） |
| **3** | **建立 Red/Green TDD 用例** | 编写测试用例，先失败(Red)后通过(Green) |
| **4** | **自动化测试** | 运行 `npm test` 直至全部通过 |
| **5** | **创建 GM 指令** | 添加调试指令协助手工测试 |
| **6** | **提交并重启** | 重启服务端/客户端，人工跟进测试 |

---

## 四、文档首次创建确认

若卡牌文档为**首次创建**，需经过人工确认后再进入开发阶段：

1. 展示完整的需求分析文档
2. 确认逻辑无误
3. 确认风险已评估
4. 获得确认后开始编写代码

若文档**已存在**，直接读取并进入开发执行阶段。
